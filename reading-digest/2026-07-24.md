# Reading digest — 2026-07-24

Found 7 paper(s) in the last 48h window.

## 1. AREX: Towards a Recursively Self-Improving Agent for Deep Research
- **Authors:** Shuqi Lu, Chaofan Li, Kun Luo, et al.
- **Summary:** AREX pairs an inner research loop that gathers evidence with an outer self-improvement loop that audits the current answer constraint-by-constraint and launches targeted follow-up research. To survive long horizons it learns an autonomous context-update tool that compresses growing interaction history into a compact "improvement state" preserving verified evidence and unresolved constraints, avoiding reliance on any external summarizer. A 4B dense and 122B-A10B MoE instantiation, trained with agentic mid-training plus long-horizon RL, are competitive with much larger models on BrowseComp, WideSearch, DeepSearchQA, and HLE.
- **Theme:** Memory for LLMs and agents (also multi-agent / agentic-system)
- **Relevance:** Directly attacks the "context grows without bound" failure mode of deep-research agents with a learned, self-managed memory rather than a scripted summarizer — and shows the recipe transfers to a real MoE at scale.
- **Link:** https://arxiv.org/abs/2607.21461

## 2. Sample-Efficient Learning from Agent Experience
- **Authors:** Chenhui Gou, Haoqin Tu, Yunhao Fang, et al.
- **Summary:** The authors frame "Experience Distillation" as the problem of turning an agent's rollout history into weight-level knowledge without any additional environment interaction. Their method retains at least 64.8% of the gains from in-context learning across 749 software-engineering tasks and six text-adventure games, whereas naive SFT on the same experience recovers only 3.8%. Against classical RL baselines it matches performance with at least 9.6× fewer environment samples.
- **Theme:** Memory for LLMs and agents (also core LLM / fine-tuning)
- **Relevance:** Concretely bridges episodic memory and parametric memory — a clean answer to "how do agents remember what they learned once the context window drops it."
- **Link:** https://arxiv.org/abs/2607.21051

## 3. NVIDIA Object-Oriented Agents (NOOA): Native Python Object-Oriented Agents
- **Authors:** Paul Furgale, Severin Klingler, James Nolan, et al.
- **Summary:** NOOA reframes an agent as a plain Python object: fields are its state, methods are its actions, docstrings are its prompts, type annotations are contracts, and a method body of `...` is filled in at runtime by an LLM-driven loop. The paper identifies six model-facing design choices — typed I/O, pass-by-reference over live objects, code-as-action, programmable loop engineering, explicit object state, and model-callable harness APIs — that together give agents and developers a single Pythonic surface. Evaluations on SWE-bench Verified, Terminal-Bench 2.0, and ARC-AGI-3 show current models use the interface effectively.
- **Theme:** Multi-agent frameworks & orchestration
- **Relevance:** Fresh datapoint on what an agent-framework abstraction should look like when it takes the "code-as-action" and "state-as-object" ideas seriously — and it ships long-term memory as a first-class field on the object.
- **Link:** https://arxiv.org/abs/2607.20709

## 4. WorldWeaver: Streaming Multi-Agent Autoregressive Diffusion with World State Registers
- **Authors:** Sicheng Mo, Yuheng Li, Ziyang Leng, Krishna Kumar Singh, Bolei Zhou
- **Summary:** WorldWeaver adds cross-agent "world state registers" — learnable tokens that persist shared world information and per-agent status across chunks — to a streaming autoregressive video diffusion model, so multi-agent rollouts don't lose logical consistency once the observation history slides out of context. The registers are grounded with supervision from agent-status labels, bird's-eye views, and scene text, and a Mixture-of-Transformers routes visual-frame modeling and world-state modeling to different weights. In two-agent Minecraft experiments, explicit world-state modeling improves both consistency and generation quality over autoregressive baselines.
- **Theme:** Multi-agent frameworks & orchestration (also memory / shared state)
- **Relevance:** A concrete architectural pattern for keeping multi-agent state coherent across rollouts by promoting shared state to a first-class token stream — a memory idea multi-agent LLM systems could borrow.
- **Link:** https://arxiv.org/abs/2607.21594

## 5. LLMs Get Lost in Evolving User Intent
- **Authors:** Jihoon Tack, Philippe Laban, Jennifer Neville
- **Summary:** The paper introduces a framework that converts any static single-turn benchmark into a dynamic multi-turn conversation whose user intent is incrementally revealed, revised, and sometimes redirected mid-conversation, while preserving the original grader. Across multiple tasks and model families, strong single-turn scores fail to transfer: models consistently lose ground once intent is allowed to evolve. The result quantifies a gap that static evaluation cannot see and that will bite any deployed collaborative agent.
- **Theme:** Core LLM advances (agentic-system evaluation)
- **Relevance:** A cheap, benchmark-agnostic protocol for turning existing datasets into dynamic-intent testbeds — closes a real hole in how we score assistant models.
- **Link:** https://arxiv.org/abs/2607.20734

## 6. Tencent WorkBuddy Bench: A Multi-Domain Coding-Agent Benchmark with Contamination-Resistant Task Construction
- **Authors:** Tencent WorkBuddy Bench Team, Siqi Cai, Shaopeng Chen, et al.
- **Summary:** WorkBuddy Bench reverse-engineers each task from a real commit, PR, or business scenario and rewrites it as a short colloquial role-played request, so the prompt is not recoverable by web-searching the underlying artifact — contamination resistance rests on this construction plus dataset versioning rather than secrecy. Four subsets (repo engineering, front-end, office/business workflows, red/blue-team security) run under a uniform task-directory format on multiple agent harnesses (CodeBuddy Code and Claude Code). The full release (tasks, environments, tests, reference solutions) makes the suite reproducible and auditable end-to-end.
- **Theme:** Core LLM advances (agentic-system evaluation)
- **Relevance:** Actively engineers around benchmark contamination, which most coding-agent benchmarks quietly ignore — and evaluates across genuinely different work domains rather than one flavor of GitHub issue.
- **Link:** https://arxiv.org/abs/2607.20911

## 7. ICAE-Bench: Evaluating Coding Agents as Interactive Project Builders
- **Authors:** Zhongyuan Peng, Dan Huang, Chuyu Zhang, et al.
- **Summary:** ICAE-Bench starts each task from a fuzzy product requirement and drives it through an automated User Agent that reveals hidden constraints only when asked, forcing the coding agent to plan, clarify, use tools, and build a full repository rather than complete a fully specified snippet. To keep the setting evaluable, tasks are anchored to precise real open-source repositories with executable behavior, and the User Agent's disclosures come from grounded "User Agent Data" so it cannot invent new requirements. Scoring combines standardized black-box tests with multi-dimensional diagnostics for functional correctness, semantic and API similarity, structural fidelity, design quality, and interaction quality.
- **Theme:** Core LLM advances (agentic-system evaluation)
- **Relevance:** Complements WorkBuddy by shifting the axis from contamination to *interaction*: the agent must negotiate ambiguity, not just execute a spec. Rare setup that grades how an agent asks as well as how it codes.
- **Link:** https://arxiv.org/abs/2607.21217
