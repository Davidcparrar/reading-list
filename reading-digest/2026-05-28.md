## 1. Is Agent Memory a Database? Rethinking Data Foundations for Long-Term AI Agent Memory
**Authors:** (arXiv 2605.26252)

**Summary:** The paper argues that long-term agent memory should be treated as a first-class data-management workload rather than a bolt-on vector store, and it diagnoses four recurring failure modes of today's memory stacks: unregulated growth, missing semantic revision, capacity-driven forgetting, and read-only retrieval. It proposes **Governed Evolving Memory (GEM)**, which models memory as evolving *state* manipulated by four state-level operators — ingestion, revision, forgetting, and retrieval — subject to six correctness conditions that constrain how that state may legally change. The contribution is conceptual/architectural: a formal data-foundations framing (operators plus correctness invariants) meant to make agent memory governable and auditable instead of an ad-hoc append log.

**Theme:** 1 — Memory for LLMs and agents (long-term memory / state management).

**Relevance:** Directly reframes the foundations of agent memory with a concrete operator + correctness-condition framework; a novel methods/architecture contribution squarely in the top-priority theme.

**Link:** https://arxiv.org/abs/2605.26252

## 2. Your Agents Are Aging Too: Agent Lifespan Engineering for Deployed Systems
**Authors:** (arXiv 2605.26302)

**Summary:** This work observes that deployed LLM agents can lose reliability over time *even with frozen model weights*, because their accumulated memory state drifts and compounds errors across sessions — making reliability a property of the whole agent harness, not the model alone. It introduces **AgingBench**, a longitudinal benchmark that not only detects degradation but classifies its *form* across four mechanisms (compression aging, interference aging, revision aging, and maintenance aging) and localizes where in the memory pipeline repair should target. The empirical finding is that aging is multi-dimensional, often invisible to standard one-shot behavioral tests, and that its locus migrates along the memory pipeline as model capability increases.

**Theme:** 1 — Memory for LLMs and agents (long-term/working-memory state management; with an agentic-system-evaluation angle).

**Relevance:** A novel longitudinal benchmark that isolates *how* agent memory degrades in deployment — a fresh, method-rich take on the central memory theme rather than another retrieval tweak.

**Link:** https://arxiv.org/abs/2605.26302

## 3. From Knowing to Doing: A Memory-Controlled Benchmark for LLM Trading Agents on Stock Markets
**Authors:** (arXiv 2605.28359)

**Summary:** **KTD-Fin** is a stock-trading benchmark built to separate what an LLM agent has *memorized* from what it can actually *reason* about, addressing two confounds in prior finance evaluations: backtest windows overlapping the model's knowledge cutoff, and raw returns being a noisy proxy for genuine skill. It applies a data-side masking protocol that anonymizes tickers and calendar information across prompts and tools, and pairs this with a Barra-style attribution that decomposes returns into market, style, and stock-selection alpha. Evaluating ten frontier agents on the Chinese CSI300 over a 2024–2026 window, the authors find masking shifts agents toward anonymized factor-based reasoning, and that their cumulative returns are largely explained by passive market/style exposure with little persistent stock-selection alpha.

**Theme:** 1 — Memory for LLMs and agents (disentangling memorization from reasoning / memory-controlled evaluation).

**Relevance:** A clean, memory-controlled evaluation design (data-side masking + attribution) that targets a core memory-vs-reasoning confound; the freshest paper in the window (~May 28) and a useful evaluation methodology contribution.

**Link:** https://arxiv.org/abs/2605.28359

## 4. AgensFlow: A Coordination-Policy Substrate for Multi-Agent Systems
**Authors:** Nicole Koenigstein

**Summary:** AgensFlow reframes multi-agent coordination as an online policy-learning problem under partial observability, so that decisions usually fixed at pipeline-design time — which skill/protocol to invoke, which role handles a subtask, which model binds to each role, and how agents are wired together — become observable, learnable choices optimized from repeated trajectories. The open-source substrate exposes use-case-conditioned skill selection, model-role assignment, topology choice, and reward auditing as a single learnable coordination policy. On distributed-systems incident and security-advisory task corpora, learned routing reaches a higher-quality operating point than a fixed pipeline on coordination-heavy classes, topology compression (`skip:X`) is shown to contribute meaningfully, and warm-started policy graphs cut exploration cost while preserving plateau quality.

**Theme:** 2 — Multi-agent frameworks & orchestration (role/model/tool assignment, coordination, topology).

**Relevance:** The strongest in-window multi-agent paper: it makes the orchestration choices (roles, models, topology, skills) explicit and learnable, with empirical results on two task corpora — a direct method contribution to theme 2.

**Link:** https://arxiv.org/abs/2605.27466

## 5. Experiments in Agentic AI for Science
**Authors:** (arXiv 2605.26305)

**Summary:** This is an experience/systems report on building autonomous agentic-AI workflows for scientific tasks using a hybrid "Local Body, Remote Brain" architecture that pairs local execution components with cloud LLM backends. It describes concrete agent instances such as DeepTS/DeepCollector for automating time-series dataset curation and DeepScribe for analyzing physics lectures. The contribution is architectural and practical — patterns for splitting agent responsibilities between local tooling and remote reasoning across multiple cooperating components — rather than a new algorithm or benchmark.

**Theme:** 2 — Multi-agent frameworks & orchestration (agent architectures; local/remote component split).

**Relevance:** A second in-window multi-agent entry that documents a concrete cooperating-agent architecture in a demanding (scientific-workflow) setting; included for theme coverage, though it is more an experience report than a methods advance, hence the lower rank.

**Link:** https://arxiv.org/abs/2605.26305

## 6. JobBench: Aligning Agent Work With Human Will
**Authors:** (arXiv 2605.26329)

**Summary:** JobBench is an agent-evaluation benchmark that deliberately targets the work people *want delegated* rather than the work of highest economic value, built on top of the Workbank survey (1,500+ workers reporting which duties they would prefer AI to automate) and spanning 130 tasks across 35 occupations chosen at the intersection of high delegation preference and high economic exposure. Each task is delivered as a workspace of heterogeneous reference files and graded by fact-anchored rubrics averaging ~35.6 binary criteria. Across 36 evaluated models the strongest (Claude Opus 4.7 under Claude Code) reaches only 45.9%, underscoring a large gap between current agents and the work humans actually prefer to delegate.

**Theme:** 3 — Core LLM advances relevant to agents (agentic-system evaluation).

**Relevance:** A carefully constructed, human-preference-grounded agent benchmark (36 models, expert-derived rubrics) that fits theme 3's agentic-system-evaluation criterion; the clearest core-LLM/eval contribution found in the window.

**Link:** https://arxiv.org/abs/2605.26329
